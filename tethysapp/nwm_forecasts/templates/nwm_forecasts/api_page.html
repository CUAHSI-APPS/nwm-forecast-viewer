    A Developers API is provided for those who wish to incorporate the National Water Model data into their own separate application or script.

    Rate Limiting and API Token

    Starting on 2018-04-20 release, anonymous/unauthenticated API calls are subject to stricter rate limiting. A HTTP status code 429 will be returned if too many API requests are made to the app once certain limit is exceeded. For example, anonymous API calls on GetWaterML endpoint are only allowed at a rate no greater than 6/min or 180/day. Authenticated API call (with a valid API Token in HTTP request header) are subject to less restricted rate limiting. For example, the rates of authenticated API calls on GetWaterML endpoint are 60/min and 1800/day.
    We encourage all developers to use API Token ASAP as anonymous/unauthenticated API calls will be fully forbidden in near future.

    Avoid Hitting the Limit

    There are two common ways to avoid hitting the rate limit.
    1. Use API Token to make authenticated API calls as it gives you more allocation;
    2. Understand how rate limiting works and make arrangement to control api call frequency. Current implementation of rate limiting uses a Rolling Window approach (not to be confused with 40-day rolling window data archive, rolling window approach is a rate limiting algorithm). For example, a endpoint has limit 6/min. Any call made to this endpoint will be blocked if 6 successful calls have already occurred in the preceding 60 seconds. So the window is rolling as new api call occurs.

    Obtain API Token

    Developer should sign in Tethys portal with a valid HydroShare account. Click USER_NAME in the upper-right corner and select "User Settings" to open user's profile page. The API Token is listed as "API Key". API Token should be kept safe.

    1. Sign in https://hs-apps.hydroshare.org with HydroShare account
    2. Click USER_NAME in the upper-right corner and select "User Settings"
    3. Copy the "API Key" and put it your code

    Add API Token to HTTP request header

    API Token should be included in http request header in the following format.
    Header Name: Header Value
    Authorization: Token API_TOKEN
    If use requests module in Python:

            r = requests.get("https://.......api_endpoint/",
                         headers={'Authorization': 'Token YOUR_API_TOKEN'},
                        )
        *Replace YOUR_API_TOKEN with a real token you got from user profile page

    Subsetting API has been changed to work asynchronously, which contain 3 endpoints:
    1) Submit a subsetting job:
    Send a HTTP POST Request with the following JSON payload to /apps/nwm-forecasts/api/submit-subsetting-job/. 
        # prepare the data sent to api endpoint
    JSON_payload = {
        'archive': 'rolling'  # optional, Accepted value: rolling, harvey, irma; Default is rolling
        'subset_parameter': {
            'config': "analysis_assim",  # AA: analysis_assim, SR: short_range, MR: medium_range, LR: long_range
            'startDate': "2017-09-20",  # YYYY-MM-DD for SR, MR & LR; Also set 'endDate' for AA
            'endDate': "2017-09-29",  # YYYY-MM-DD (only for AA)
            'geom': "channel_rt",  # forcing, channel_rt, reservoir, land
            'time': "00",  # 00, 01 ... 23 (only for SR, MR and LR)
            'mem': "1",  # 1, 2, 3, 4 (LR ensemble member)
            'merge': True  # True or False
        },
        'watershed_epsg': 4326,  # keep this unchanged as latest GeoJSON standard requires coordinates in WGS84/EPSG:4326 projection
                                 # see: https://tools.ietf.org/html/rfc7946#section-4
        'watershed_geometry': geojson_polygon_str_4326,  # the polygon geojson string or a bounding box json string in WGS84/EPSG:4326
        'bbox': False,  # Optional, default: False; Set to True if watershed_geometry is a bounding box json string
    }

    Expected response:
    {"job_id": JOB_ID, "status": JOB_STATUS} 
    Get "latest" data without specifying "startDate", "endDate" or "time" (only works when archive='rolling')

For AA:
    a) Subset data starting from t00z of day "YYYY-MM-DD" to Today(UTC)'s latest data
    "startDate": "YYYY-MM-DD"
    "endDate": "latest

    b) Subset data starting from t00z of the Xth day before Today to Today's latest data
    "startDate": "X" # X is a number indicates the Xth day before today
    "endDate": "latest

    c) Subset latest data for Today(UTC) starting from t00z
    "startDate": ""
    "endDate": "latest"

For SR, MR, LR:
    Subset latest timestamp for Today(UTC), and parameter "time" is ignored
    "startDate": "latest"

    2) Check job status:
    Send a HTTP GET Request with parameter "job_id" to /apps/nwm-forecasts/api/check-subsetting-job-status/?job_id=JOB_ID.
    Expected response:
    {"status": JOB_STATUS}
    If the value of 'status' is 'SUCCESS', it means the job is done.

    3) Retrieve subetting results:
    Send a HTTP GET Request with parameter "job_id" to /apps/nwm-forecasts/api/download-subsetting-results/?job_id=JOB_ID.
    Expected response: subsetting results in one zip file
